# SystemVerilog Support Design Document

## How the AST Works

All AST C++ files are **generated code**. The single source of truth is:

```
lib/tools/ASTGen/verilog_ast.yaml
```

`astgen.py` reads it and uses Jinja2 templates to emit:
- `lib/include/veriparse/AST/*.hpp` — one header per node
- `lib/src/AST/*.cpp` — one impl per node
- Plus: `nodes.hpp`, `node_type.*`, `node_to_node_type.hpp`, `node_ostream.*`,
  all generator/importer glue

**To add an AST node:** add an entry to `verilog_ast.yaml`, re-run `astgen.sh`.
The YAML schema is simple:

```yaml
MyNode:
  inherit: ParentNode          # optional, default is Node
  properties:
    name: "std::string"
    kind: enum(FOO, BAR, BAZ)  # generates a C++ enum MyNode::KindEnum
    flag: bool
  children:
    item: OtherNode            # single child → OtherNode::Ptr
    items: list(OtherNode)     # list child  → OtherNode::ListPtr
```

## SV Mode Flag

**Required, not boilerplate.**

SV keywords like `logic`, `always_ff`, `always_comb` must be gated behind a
flag. Without it, any Verilog-2001 design using those as identifiers (legal in
V2001) would silently break. Every major tool gates SV parsing this way
(Icarus `-g2012`, Verilator `--sv`, Yosys `read_verilog -sv`).

The flag gates scanner keyword recognition and SV-specific grammar rules.
Implementation: a `bool sv_mode` threaded into `VerilogScanner` and
`VerilogParser` at construction time, exposed as a CLI flag and API parameter.

---

## Design Principle: Preserve SV Constructs

veriparse is a **source-to-source** tool. SV constructs must be preserved
as-is in the output so users can verify equivalence and the result remains
valid SV. This means:

- **No lowering** of SV constructs to Verilog-2001 equivalents
- Each SV construct gets its **own AST node(s)**
- The existing transformation passes operate on the AST generically and
  remain unmodified — they already walk `Node::Ptr` trees, so new node types
  are handled transparently (visited but not specially transformed)
- The **YAML generator / YAML importer** and **dot generator** are
  auto-regenerated by `astgen.sh` and cover new nodes for free
- Only the **Verilog pretty-printer** (if one exists) needs extending for
  new nodes; otherwise the YAML output is the interchange format

## What to Add — Prioritized

### Phase 1 — New AST nodes + enum extensions for synthesizable SV

#### `parameter` / `localparam` with SV types

No new node needed. The existing `Parameter.type` enum is extended:

```yaml
# verilog_ast.yaml
Parameter:
  properties:
    name: "std::string"
    sign: bool
    type: enum(INTEGER, REAL, LOGIC, INT, BIT, BYTE, SHORTINT, LONGINT, NONE)
  children:
    value: Node
    widths: list(Width)
```

`Localparam` inherits `Parameter` and gets the new values for free.

In the parser, `param_type` / `localparam_type` get new alternatives gated on
`sv_mode` for `TK_INT`, `TK_BIT`, `TK_LOGIC`, `TK_BYTE`, `TK_SHORTINT`,
`TK_LONGINT` — each sets the appropriate `TypeEnum` value and optional widths.

#### Net / variable types

| SV Construct | New AST node | Inherits | Notes |
|---|---|---|---|
| `logic [N:0] name` | `Logic` | `Net` | Same structure as `Wire`/`Reg` |

#### Always variants

| SV Construct | New AST node | Inherits | Notes |
|---|---|---|---|
| `always_ff @(...)` | `AlwaysFF` | `Always` | Same children (senslist + statement) |
| `always_comb` | `AlwaysComb` | `Always` | No senslist (implicit `*`) |
| `always_latch` | `AlwaysLatch` | `Always` | No senslist |

#### Case variants

| SV Construct | New AST node | Inherits | Notes |
|---|---|---|---|
| `unique case` | `UniqueCaseStatement` | `CaseStatement` | No extra children |
| `priority case` | `PriorityCaseStatement` | `CaseStatement` | No extra children |

#### Attribute in generate blocks

No new node. Parser fix only: add `pragmalist generate_item` production
(one grammar rule, already supported elsewhere).

### Phase 2 — Type system additions

| SV Construct | New AST node(s) | Notes |
|---|---|---|
| `typedef logic [N:0] name` | `Typedef` — property: name; children: type Node, widths | |
| `enum { A=0, B } name` | `EnumDef`, `EnumItem` — EnumItem: name + optional value Node | |
| `typedef enum ... name` | `Typedef` wrapping `EnumDef` | |
| packed `struct` | `StructDef`, `StructMember` | |

### Phase 3 — Larger effort (out of scope for synthesizable subset)

- `interface` / `modport` — new nodes + significant grammar
- `package` / `import` — new nodes
- `foreach` loop — `ForeachStatement` node

## Impact on Existing Passes

The transformation passes (`generate_removal`, `module_flattener`,
`constant_folding`, etc.) walk the AST via `TransformationBase::process` which
dispatches on `Node::Ptr`. New node types are traversed as generic `Node`
children — **no pass modification required** for the nodes to survive
transformation. Passes that pattern-match on specific node types (e.g.
always sensitivity analysis) may need extending if SV `always_ff` blocks
must be treated equivalently, but this can be done incrementally.
