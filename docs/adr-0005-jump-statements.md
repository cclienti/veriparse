# ADR-0005 — SystemVerilog jump statements (`return` / `break` / `continue`)

- **Status**: Proposed — design only; not yet implemented.
- **Date**: 2026-06-28
- **Scope**: Add the SystemVerilog jump statements (IEEE 1800-2017 §12.8) —
  `return [ expression ] ;`, `break ;`, `continue ;` — end to end: scanner,
  grammar, AST, generator, **and** the transformation passes whose control-flow
  analysis they change (function evaluation, loop unrolling, dead-code).
- **Normative reference** — IEEE 1800-2017, verified against `docs/1800-2017.pdf`:
  **§12.8** (Jump statements) and its BNF (Annex A.6.5); subroutines **§13.4**.
  Jump statements are **SystemVerilog-only** — in Verilog-1364 (`docs/verilog-std-1364-2005.pdf`)
  `return`/`break`/`continue` are ordinary identifiers.

---

## 1. Problem

`return`, `break`, and `continue` are **not keywords** in veriparse today — the
scanner lexes them as identifiers. Consequences:

- `return ;` / `break ;` / `continue ;` **accidentally parse** — as a neutral
  subroutine *call* to an identifier named `return`/`break`/`continue`, **not** a
  jump statement (semantically wrong; the AST is a bogus `Call`).
- `return expr ;` is a **syntax error** (`return expr` lexes as two identifiers →
  "unexpected identifier, expecting `=`").

So the entire `jump_statement` family (§12.8) is unimplemented. The forms are
common in synthesizable SV — `return expr;` in functions, `break`/`continue` in
`for` loops — and `return expr;` is the most impactful gap (the value-returning
function form).

The BNF (Syntax 12-6, from Annex A.6.5):

```
jump_statement ::= return [ expression ] ; | break ; | continue ;
```

Prose constraints (§12.8): `return` only in a subroutine (with an expression of
the correct type in a value function, bare in a task/void function); `break` and
`continue` only in a loop. SystemVerilog has no `goto`.

---

## 2. Front end — scanner / grammar / AST / generator

### 2.1 Scanner — keywords gated on `sv_mode`
Add `return` / `break` / `continue` as keyword tokens (`TK_RETURN`, `TK_BREAK`,
`TK_CONTINUE`), **gated on SV mode** like the other SV-only keywords — in Verilog
mode they must stay identifiers (1364 back-compat). The scanner already has the
SV-mode keyword gate; these slot into it.

### 2.2 Grammar
A statement-level production:

```
jump_statement : TK_RETURN expression TK_SEMICOLON   -> Return{ value: expr }
               | TK_RETURN TK_SEMICOLON              -> Return{ value: null }
               | TK_BREAK  TK_SEMICOLON              -> Break
               | TK_CONTINUE TK_SEMICOLON            -> Continue
```

Added to the procedural `statement` alternatives. The parser stays **permissive**
— it does *not* enforce "return only in a subroutine / break-continue only in a
loop"; that is a §12.8 semantic constraint for an optional later validation pass,
not a syntactic one.

### 2.3 AST — node per form (ADR-0001 principle: discriminate by node type)
Following ADR-0001 (no `kind` enum; a distinct node per form):

```yaml
Return:    { children: { value: Node } }   # value null for bare `return;`
Break:     {}                              # leaf
Continue:  {}                              # leaf
```

Three statement nodes. (Rejected: one `Jump { kind: enum, expr }` — a `kind` enum
is exactly what ADR-0001 §3 argues against.) `Break`/`Continue` are childless
leaves; `Return` carries the optional value expression.

### 2.4 Generator
`render_return` → `return [<value>];`; `render_break` → `break;`;
`render_continue` → `continue;`. Hand-written in `VerilogGenerator` (the Verilog
generator is hand-maintained; the YAML/dot generators and the importer are
regenerated by `make ast-gen`).

---

## 3. Pass implications (the substance)

Jump statements introduce **non-fall-through control flow**. Most passes carry
them through untouched, but the ones that **interpret or restructure procedural
code** must respect them. The principle: *a jump changes which statements execute,
so any pass that reasons about "what executes" must model it.*

### 3.1 FunctionEvaluation — `return` is an early exit (control-flow signal)
`FunctionEvaluation` interprets a **constant function** to fold a call to a
constant. Today the function's result is the value left in the function-name
variable, read after the body runs straight through. `return expr` is an
**explicit early exit**, so the interpreter must gain control-flow signals:

- `return expr` → set the result, **unwind** (stop executing the remaining body);
- `return` (void) → unwind;
- `break` / `continue` inside an interpreted loop → end / skip-to-next iteration.

Model it as a status threaded through statement evaluation —
`enum Flow { NORMAL, RETURNED, BROKE, CONTINUED }` (or an unwind exception): each
compound statement checks the child's status and stops/propagates accordingly.
**Backward compatible**: the legacy `name = expr;` form still works — falling off
the end is an implicit `return name`. This is the single most important pass
change: without it a const function that uses `return` cannot be folded.

### 3.2 LoopUnrolling — lower `break` / `continue`
Static unrolling of a constant-bound loop must preserve jump semantics. Per the
loop body:

- **No jumps** → unchanged (today's behaviour).
- **`continue`** → the statements *after* the continue in that iteration are
  skipped. Lower to a guard over the remainder of the iteration body
  (`if (!taken_i) begin <rest> end`); when the continue condition is
  **compile-time constant**, drop the rest statically with no residual guard.
- **`break`** → all *subsequent* iterations are skipped. Lower to an **"active"
  guard** threaded across the unrolled iterations (a `broke` flag; each unrolled
  iteration becomes `if (!broke) begin <iter> end`, and `break` sets it); when the
  break condition is **compile-time constant**, stop emitting iterations statically.

Constant-condition jumps resolve to static cut-offs (clean output, no flags);
runtime-condition jumps need the guard/flag lowering. **Phase-1 fallback**: only
unroll loops whose jump conditions are statically determinable, and emit a clear
diagnostic for the rest, deferring the general flag lowering.

### 3.3 DeadcodeElimination — unreachable after a jump
Statements that follow an **unconditional** `return`/`break`/`continue` within the
same statement block are unreachable and may be removed. Cheap, and it keeps the
output of §3.1/§3.2 tidy. (Conditional jumps — under an `if` — do **not** make the
following statements dead.)

### 3.4 ConstantFolding
No direct change. A constant function call folds via FunctionEvaluation (§3.1);
folding of ordinary expressions is unaffected (a jump is a statement, not an
operand).

### 3.5 Pass-through — ModuleFlattener, io/instance normalizers, obfuscator, …
Jump statements in ordinary procedural blocks (`always`/`initial`) that are **not**
compile-time-evaluated are carried to the output **unchanged** — the downstream
synthesis/simulation tool implements them. `ModuleFlattener` clones them like any
statement (works for free once the AST node exists); `ModuleObfuscator` renames
identifiers only and a jump node has none (`Return.value` is an expression it
already recurses into).

### 3.6 Validation (optional, future)
A small checker could enforce the §12.8 placement rules (return ⊂ subroutine,
break/continue ⊂ loop) now that the parser is permissive. Out of scope for the
first implementation.

---

## 4. Phasing

1. **Front end** — scanner keywords (SV-gated) + grammar + `Return`/`Break`/
   `Continue` AST nodes + generator. Round-trips `return [expr];`/`break;`/
   `continue;`. Parser/generator golden tests.
2. **FunctionEvaluation** — return/break/continue control flow in the interpreter
   (§3.1). Unblocks constant functions written with `return`.
3. **LoopUnrolling** — break/continue lowering (§3.2), constant-condition first,
   then the general guard/flag form.
4. **DeadcodeElimination** — unreachable-after-jump (§3.3). Optional.

Each phase lands green on its own; phase 1 is independently useful (parse + emit).

---

## 5. Out of scope

- `disable` (§13.5, named-block disable) — a different control construct.
- `break`/`continue` outside loops, `return` outside subroutines — illegal SV; a
  validation concern (§3.6), not parsed-away.
- `fork`/`join` interactions with jumps (§12.8 note) — not synthesizable.
- Non-blocking/event control intertwined with jumps — handled by carrying the
  statements through (§3.5), not by evaluating them.
