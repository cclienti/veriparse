# ADR-0005 ‚Äî SystemVerilog jump statements (`return` / `break` / `continue`)

- **Status**: Accepted ‚Äî implemented (front end + passes, incl. the conservative-
  interpretation pivot of ¬ß3.1.1), **except** the ¬ß3.2.1 flag lowering for
  mixed/nested loop jumps, which is designed but **not implemented**: those loops
  are left intact with a diagnostic.
- **Date**: 2026-06-28 (amended 2026-07-02: ¬ß3.1.1 conservative interpretation)
- **Scope**: Add the SystemVerilog jump statements (IEEE 1800-2017 ¬ß12.8) ‚Äî
  `return [ expression ] ;`, `break ;`, `continue ;` ‚Äî end to end: scanner,
  grammar, AST, generator, **and** the transformation passes whose control-flow
  analysis they change (function evaluation, loop unrolling, dead-code).
- **Normative reference** ‚Äî IEEE 1800-2017, verified against `docs/1800-2017.pdf`:
  **¬ß12.8** (Jump statements) and its BNF (Annex A.6.5); subroutines **¬ß13.4**.
  Jump statements are **SystemVerilog-only** ‚Äî in Verilog-1364 (`docs/verilog-std-1364-2005.pdf`)
  `return`/`break`/`continue` are ordinary identifiers.

---

## 1. Problem

`return`, `break`, and `continue` are **not keywords** in veriparse today ‚Äî the
scanner lexes them as identifiers. Consequences:

- `return ;` / `break ;` / `continue ;` **accidentally parse** ‚Äî as a neutral
  subroutine *call* to an identifier named `return`/`break`/`continue`, **not** a
  jump statement (semantically wrong; the AST is a bogus `Call`).
- `return expr ;` is a **syntax error** (`return expr` lexes as two identifiers ‚Üí
  "unexpected identifier, expecting `=`").

So the entire `jump_statement` family (¬ß12.8) is unimplemented. The forms are
common in synthesizable SV ‚Äî `return expr;` in functions, `break`/`continue` in
`for` loops ‚Äî and `return expr;` is the most impactful gap (the value-returning
function form).

The BNF (Syntax 12-6, from Annex A.6.5):

```
jump_statement ::= return [ expression ] ; | break ; | continue ;
```

Prose constraints (¬ß12.8): `return` only in a subroutine (with an expression of
the correct type in a value function, bare in a task/void function); `break` and
`continue` only in a loop. SystemVerilog has no `goto`.

---

## 2. Front end ‚Äî scanner / grammar / AST / generator

### 2.1 Scanner ‚Äî keywords gated on `sv_mode`
Add `return` / `break` / `continue` as keyword tokens (`TK_RETURN`, `TK_BREAK`,
`TK_CONTINUE`), **gated on SV mode** like the other SV-only keywords ‚Äî in Verilog
mode they must stay identifiers (1364 back-compat). The scanner already has the
SV-mode keyword gate; these slot into it.

### 2.2 Grammar
A statement-level production:

```
jump_statement : TK_RETURN expression TK_SEMICOLON   -> Return{ value: expr }
               | TK_RETURN TK_SEMICOLON              -> Return{ value: null }
               | TK_BREAK  TK_SEMICOLON              -> Break
               | TK_CONTINUE TK_SEMICOLON            -> Continue
```

Added to the procedural `statement` alternatives. The parser stays **permissive**
‚Äî it does *not* enforce "return only in a subroutine / break-continue only in a
loop"; that is a ¬ß12.8 semantic constraint for an optional later validation pass,
not a syntactic one.

### 2.3 AST ‚Äî node per form (ADR-0001 principle: discriminate by node type)
Following ADR-0001 (no `kind` enum; a distinct node per form):

```yaml
Return:    { children: { value: Node } }   # value null for bare `return;`
Break:     {}                              # leaf
Continue:  {}                              # leaf
```

Three statement nodes. (Rejected: one `Jump { kind: enum, expr }` ‚Äî a `kind` enum
is exactly what ADR-0001 ¬ß3 argues against.) `Break`/`Continue` are childless
leaves; `Return` carries the optional value expression.

### 2.4 Generator
`render_return` ‚Üí `return [<value>];`; `render_break` ‚Üí `break;`;
`render_continue` ‚Üí `continue;`. Hand-written in `VerilogGenerator` (the Verilog
generator is hand-maintained; the YAML/dot generators and the importer are
regenerated by `make ast-gen`).

## 3. Pass implications (the substance)

Jump statements introduce **non-fall-through control flow**. Most passes carry
them through untouched, but the ones that **interpret or restructure procedural
code** must respect them. The principle: *a jump changes which statements execute,
so any pass that reasons about "what executes" must model it.*

### 3.1 FunctionEvaluation ‚Äî `return` is an early exit (control-flow signal)
`FunctionEvaluation` interprets a **constant function** to fold a call to a
constant. Today the function's result is the value left in the function-name
variable, read after the body runs straight through. `return expr` is an
**explicit early exit**, so the interpreter must gain control-flow signals:

- `return expr` ‚Üí set the result, **unwind** (stop executing the remaining body);
- `return` (void) ‚Üí unwind;
- `break` / `continue` inside an interpreted loop ‚Üí end / skip-to-next iteration.

Model it as a status threaded through statement evaluation ‚Äî
`enum Flow { NORMAL, RETURNED, BROKE, CONTINUED }` (or an unwind exception): each
compound statement checks the child's status and stops/propagates accordingly.
**Backward compatible**: the legacy `name = expr;` form still works ‚Äî falling off
the end is an implicit `return name`. This is the single most important pass
change: without it a const function that uses `return` cannot be folded.

#### 3.1.1 Conservative interpretation ‚Äî commit only what is fully interpreted

The first implementation tried to keep *partial* results when the interpreter hit
something it could not resolve: fold what executed, then *invalidate* the state
the skipped construct might have touched (erase the variables its body assigns),
plus a reachability scan for a `return` left under a non-constant condition.
That soundness model is a **blacklist that must be exhaustive**: a `Return`
assigns no variable so the return target survived invalidation; a bit-select
lvalue (`f[0] = ‚Ä¶`) is a `Pointer`, not an `Identifier`, so the base variable
survived; a loop whose *initial* condition was non-constant never reached the
invalidation call at all. Each hole is silent wrong code ‚Äî a folded constant that
disagrees with simulation. Three generations of such holes were patched before
the model itself was rejected.

The pivot inverts it. `VariableFolding` tracks a single **fully-interpreted**
flag, and any construct it cannot resolve to concrete control flow and concrete
values triggers one uniform reaction ‚Äî *give up*:

- **all interpreter state is dropped** (the whole value map, not a computed
  subset ‚Äî there is no per-construct erase list to keep exhaustive), so nothing
  downstream can fold to a stale value; and
- the run is marked **not fully interpreted**.

Give-up triggers: an `if` whose condition does not fold to a constant; a loop
whose condition/bound does not fold at any point (or that exceeds the
configurable unroll cap); an unrolled loop whose body still holds an unresolved
jump (left intact for ¬ß3.2); a task or unresolved call statement (unknown side
effects ‚Äî its arguments are also left **untouched**, since without the callee's
declaration an actual may be an output writeback target that folding to a
constant would destroy); a write through an lvalue whose base variable cannot be
identified; and **any statement kind the interpreter does not model** (case,
delays, event controls, `forever`, ‚Ä¶) ‚Äî unknown-by-default, not
known-by-default. Two precise
(non-give-up) rules complete the model: an rvalue that does not fold *erases its
target* from the state (never keeps the stale value), and a partial-value write
(`f[0] = ‚Ä¶`) erases the whole base variable.

`FunctionEvaluation` then folds a call **only when the run stayed fully
interpreted**; otherwise it leaves the call un-evaluated (nullptr). This
replaces the data-dependent-conditional-return scan this section previously
described (deleted): instead of proving after the fact that no unresolved
`return` escaped, the interpreter refuses to certify anything it did not fully
execute. The precision loss is nil for the intended use ‚Äî a function worth
folding is a constant function, and a constant function is fully interpretable
by definition.

Two interpreter-fidelity rules land with the pivot: the `for` interpreter tests
the condition **before** the first iteration (a zero-trip loop executes
nothing ‚Äî the previous do-while shape ran the body once), and an iteration that
consumes a jump **truncates at execution time** the statements the jump made
unreachable (a spliced branch's tail is dropped with the jump that precedes it,
so `begin break; y = 1; end` never leaks `y = 1` into the unrolled output while
later iterations of a consumed `continue` are preserved).

### 3.2 LoopUnrolling ‚Äî lower `break` / `continue`
Static unrolling of a constant-bound loop must preserve jump semantics. A cloned
`break`/`continue` would land outside any loop, so the unroller lowers `if (cond)
jump;` to a **structural guard ‚Äî no flag variable** (which avoids inventing a
declaration and keeps the output purely combinational):

The remainder after a jump is moved into the **`else` branch of the jump's own
condition** ‚Äî `if (c) jump; rest` becomes `if (c) begin end else begin rest end`.
This keeps the source's **X-propagation**: the remainder runs whenever the jump is
not taken, *including when `c` is X* (`if (x)` falls to the else). A negated guard
`if (!c) begin rest end` would instead skip the remainder on X and so diverge from
the source in simulation; the `!==` that would fix that is not synthesizable.

- **No jumps** ‚Üí unchanged.
- **`continue`** ‚Üí the statements *after* the continue in that iteration are
  skipped. Lowered into the body *before* unrolling. Each unrolled iteration then
  guards only its own remainder; later iterations are unaffected.
- **`break`** ‚Üí the rest of this iteration *and* every later one is skipped.
  Lowered *after* unrolling, across the flat iteration sequence: at each
  `if (c) break;`, everything after it (this iteration's remainder plus all later
  iterations) moves into the `else`, so the unrolled body is a chain of nested
  guards that stops at the first taken break. A bare (unconditional) jump drops the
  unreachable remainder.

Downstream constant folding collapses guards whose condition is compile-time
constant, so the constant case needs no special path.

**Implemented scope** (for and repeat loops): a single jump kind written as
`if (cond) jump;` or a bare `jump;` at the loop-body statement level ‚Äî the
synthesizable norm. Anything else is left intact with a diagnostic (valid, just
not unrolled):

- a loop mixing `break` **and** `continue`;
- a jump nested deeper ‚Äî inside a block or a compound `if`.

#### 3.2.1 Mixed / nested jumps ‚Äî why they are deferred (flag lowering)

The declaration-free `else`-branch form does **not** extend to the two deferred
cases, and the reason is structural, not incidental:

- **Mixed break+continue is exponential without a flag.** `continue` turns a
  statement sequence into a branch (`if (c) begin end else begin rest end`);
  a later `break` then has to nest *everything after it ‚Äî including all remaining
  unrolled iterations ‚Äî into both sides of every preceding continue-branch. Each
  continue roughly doubles the continuation, so N iterations with a continue blow
  up as O(2^N). The nested case has the same continuation-duplication problem.

- **A flag makes it linear** ‚Äî and stays X-faithful. Introduce `_brk` (persists
  across iterations) and `_cont` (reset each iteration); lower `break`‚Üí`_brk = 1`,
  `continue`‚Üí`_cont = 1`, guard the remainder with `if (_brk==0 && _cont==0) ‚Ä¶`,
  and wrap each unrolled iteration in `if (_brk==0) begin _cont = 0; ‚Ä¶ end`.
  Because the guard tests a *variable* (not the path condition), the remainder
  appears once ‚Äî linear, including for nested jumps. It is X-faithful because
  `if (c) _brk = 1;` leaves the flag `0` when `c` is X (break not taken ‚Üí the
  remainder runs, matching the source), and the flag is never X so `_brk==0` is
  clean. Verified by simulation: the flag form matches the source loop for every
  input, including partial-X.

- **Blocker ‚Äî where the flag is declared.** The flags need an integer
  declaration. A block-local decl does *not* work: `ScopeElevator` hoists it to a
  mid-`always` position (after existing statements), which **Verilator rejects**
  (declarations must precede statements). The declaration must therefore be placed
  at **module level** (Verilator-clean, confirmed by lint). That means the flag
  path must thread module-level decl injection through `LoopUnrolling::process`.

**Status: deferred ‚Äî not implemented.** The simple single-jump lowering above is
implemented; the flag-based path for mixed/nested jumps is a design only ‚Äî the
flag form was validated by simulation, but **no code for it exists** and such
loops are left intact (valid, just not unrolled) with a diagnostic. Revisit if a
design needs jump-containing loops of this shape actually unrolled; the work is
the flag lowering itself plus module-level declaration injection through
`LoopUnrolling::process`.

### 3.3 DeadcodeElimination ‚Äî unreachable after a jump
Statements that follow an **unconditional** `return`/`break`/`continue` within the
same statement block are unreachable and may be removed. Cheap, and it keeps the
output of ¬ß3.1/¬ß3.2 tidy. (Conditional jumps ‚Äî under an `if` ‚Äî do **not** make the
following statements dead.)

### 3.4 ConstantFolding
No direct change. A constant function call folds via FunctionEvaluation (¬ß3.1);
folding of ordinary expressions is unaffected (a jump is a statement, not an
operand).

### 3.5 Pass-through ‚Äî ModuleFlattener, io/instance normalizers, obfuscator, ‚Ä¶
Jump statements in ordinary procedural blocks (`always`/`initial`) that are **not**
compile-time-evaluated are carried to the output **unchanged** ‚Äî the downstream
synthesis/simulation tool implements them. `ModuleFlattener` clones them like any
statement (works for free once the AST node exists); `ModuleObfuscator` renames
identifiers only and a jump node has none (`Return.value` is an expression it
already recurses into).

### 3.6 Validation (optional, future)
A small checker could enforce the ¬ß12.8 placement rules (return ‚äÇ subroutine,
break/continue ‚äÇ loop) now that the parser is permissive. Out of scope for the
first implementation.

## 4. Phasing

1. **Front end** ‚úÖ ‚Äî scanner keywords (SV-gated) + grammar + `Return`/`Break`/
   `Continue` AST nodes + generator. Round-trips `return [expr];`/`break;`/
   `continue;`. Parser/generator golden tests.
2. **FunctionEvaluation** ‚úÖ ‚Äî return/break/continue control flow in the interpreter
   (¬ß3.1). Unblocks constant functions written with `return`.
3. **LoopUnrolling** ‚úÖ (simple case) ‚Äî structural `else`-branch lowering of a
   single top-level `if (cond) jump;` / bare jump, for `for` and `repeat` (¬ß3.2).
   Mixed break+continue and deeper-nested jumps are **deferred** (flag lowering,
   ¬ß3.2.1) ‚Äî left intact for now.
4. **DeadcodeElimination** ‚úÖ ‚Äî unreachable-after-jump (¬ß3.3): within a block,
   statements after an unconditional break/continue/return are dropped; a
   conditional jump leaves its siblings. Runs before the existing deadcode steps.
5. **Conservative-interpretation pivot** ‚úÖ ‚Äî ¬ß3.1.1: VariableFolding gives up
   (drops all state, marks the run not fully interpreted) on anything it cannot
   resolve; FunctionEvaluation folds only fully-interpreted runs. Replaces the
   per-construct state-invalidation approach and its reachability heuristics.
6. **Flag-based lowering for mixed/nested loop jumps** ‚úó **not implemented** ‚Äî
   ¬ß3.2.1: the flag form is designed and simulation-validated, but no code was
   written; such loops are left intact with a diagnostic.

Each phase lands green on its own; phase 1 is independently useful (parse + emit).

---

## 5. Out of scope

- `disable` (¬ß13.5, named-block disable) ‚Äî a different control construct.
- `break`/`continue` outside loops, `return` outside subroutines ‚Äî illegal SV; a
  validation concern (¬ß3.6), not parsed-away.
- `fork`/`join` interactions with jumps (¬ß12.8 note) ‚Äî not synthesizable.
- Non-blocking/event control intertwined with jumps ‚Äî handled by carrying the
  statements through (¬ß3.5), not by evaluating them.
