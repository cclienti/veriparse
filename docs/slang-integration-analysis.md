# Slang Integration Analysis for Veriparse

## Context

Veriparse is a Verilog-2005 RTL transformation toolkit built around:
- A **Flex/Bison parser** (~4200 lines)
- A **mutable AST** (~80 node types) generated by a Python/Jinja2 code generator
- A set of **transformation passes** (flattening, obfuscation, constant folding, etc.)
- A **Verilog generator** that emits source-faithful RTL output

The question explored here: can veriparse be extended to support SystemVerilog,
and what role could Slang or other tools play?

---

## Could Yosys Replace Veriparse?

### What Yosys already handles (no extension needed)

| veriparse pass          | Yosys equivalent              |
|-------------------------|-------------------------------|
| `GenerateRemoval`       | built-in elaboration          |
| `LoopUnrolling`         | built-in elaboration          |
| `ConstantFolding`       | `opt_const`                   |
| `BranchSelection`       | `opt_const` + `opt_clean`     |
| `DeadcodeElimination`   | `opt_clean`                   |
| `ModuleFlattener`       | `flatten`                     |
| `ParameterInliner`      | built-in elaboration          |

### Fundamental blocker: RTL-faithful output

Yosys's internal representation (RTLIL) is a **netlist IR**, not an AST.
`write_verilog` emits gates and wires ‚ÄĒ not readable RTL. Reconstructing
source-faithful RTL from RTLIL is extremely lossy. This is veriparse's
core value: it never leaves the AST world.

### What would need Yosys extensions

| Feature                        | Complexity    |
|--------------------------------|---------------|
| RTL-preserving `write_verilog` | ūüĒī Very High  |
| Selective parameter flattening | ūüü° Medium     |
| Seeded obfuscation pass        | ūüüĘ Low-Medium |
| RTL-level function inlining    | ūüü° Medium     |

### Verdict

Yosys could be used for **formal verification** of veriparse output
(equivalence checking via `equiv_check`) but cannot replace veriparse's
RTL transformation pipeline.

## SystemVerilog Extension Complexity

### Current parser scope

- Standard: **Verilog-2005**
- Parser: Flex/Bison (~4200 lines grammar, ~330 lines scanner)
- AST: ~80 node types defined in `verilog_ast.yaml`

### Effort breakdown for synthesizable SV subset

| Phase                                   | Effort            |
|-----------------------------------------|-------------------|
| Lexer: new SV keywords                  | ūüüĘ 1 week         |
| Parser: `logic`, `always_ff/comb`       | ūüüĘ 1-2 weeks      |
| Parser: `enum`, `typedef`, `struct`     | ūüü° 1-2 months     |
| Parser: `interface`/`modport`/`package` | ūüĒī 3-6 months     |
| New AST nodes (via ASTGen)              | ūüü° 2-4 weeks      |
| Pass updates                            | ūüĒī 3-6 months     |
| Verilog generator updates               | ūüü° 2-4 weeks      |
| **Total**                               | **9-18 months solo** |

The hardest single item is **interface/modport flattening** ‚ÄĒ comparable
in complexity to the entire existing `module_flattener.cpp`.

## Slang vs Surelog as a Reimplementation Base

### What veriparse's architecture requires

- A **mutable AST** with parent tracking
- **In-place tree rewriting** via visitor pattern
- **Node replacement** in parent context
- A **Verilog generator** that walks the AST back to source-faithful RTL

### Slang

**Strengths:**
- Best-in-class SV parser ‚ÄĒ passes almost all SV compliance tests
- Full elaboration (parameters, generate, types)
- Clean C++ API, actively maintained

**Weaknesses:**
- AST is **immutable** and compilation-oriented ‚ÄĒ not designed for rewriting
- No built-in AST mutation API
- No RTL Verilog emitter (only netlist/MLIR lowering)

**Verdict:** Excellent frontend, but transformation + emission layers
would need to be built from scratch.

### Surelog / UHDM

**Strengths:**
- Full SV parser used by OpenROAD
- UHDM is **mutable** with visitor support

**Weaknesses:**
- Post-elaboration IR ‚ÄĒ closer to netlist than RTL AST
- `uhdm2verilog` output fidelity is limited
- Complex, poorly documented API

### Comparison table

| Criterion                | veriparse (current) | Slang-based    | Surelog/UHDM   | CIRCT/MLIR       |
|--------------------------|---------------------|----------------|----------------|------------------|
| SV support               | Verilog-2005        | ‚úÖ Full SV     | ‚úÖ Full SV     | ‚úÖ Full SV       |
| Mutable AST              | ‚úÖ                  | ‚ĚĆ build own   | ‚ö†ÔłŹ partial     | ‚úÖ MLIR          |
| RTL-faithful output      | ‚úÖ                  | ‚ĚĆ             | ‚ö†ÔłŹ limited     | ‚ĚĆ               |
| Transformation API       | ‚úÖ custom visitor   | ‚ĚĆ none        | ‚ö†ÔłŹ visitor     | ‚úÖ MLIR patterns |
| Effort to reimplement    | ‚ÄĒ                   | ūüĒī Very High   | ūüĒī High        | ūüĒī Very High     |

## Proposed Approach: ASTGen Backend for Slang

### Core idea

Rather than a full rewrite, add a new **code generation layer** to the
existing `astgen.py` system that generates a **Slang‚Üíveriparse AST importer**.
The `verilog_ast.yaml` remains the single source of truth.

### Architecture

```
SV source
    ‚Ēā
    ‚Ėľ
Slang (parse + elaborate)
    ‚Ēā  Slang::AST::*  (immutable, typed, elaborated)
    ‚Ėľ
[Generated] SlangImporter        ‚Üź new ASTGen template
    ‚Ēā  converts Slang nodes ‚Üí veriparse AST nodes
    ‚Ėľ
veriparse AST (mutable)
    ‚Ēā
    ‚Ėľ
Existing transformation passes   ‚Üź unchanged
    ‚Ēā
    ‚Ėľ
Verilog generator (existing)     ‚Üź minor SV‚ÜíV2005 downgrade additions
    ‚Ēā
    ‚Ėľ
RTL Verilog output
```

### ASTGen system ‚ÄĒ current strengths

The existing code generation system (`lib/tools/ASTGen/`) is a key asset:

- **Single source of truth**: `verilog_ast.yaml` drives all C++ generation
- **Jinja2 templates** generate: node classes, type enums, YAML serialization,
  DOT visualizer, round-trip tests ‚ÄĒ all automatically
- **Inheritance support** via `inherit:` key with proper member propagation
- **Smart file writing**: unchanged files are not touched (avoids recompilation)
- **Auto-generated tests**: every new node gets free round-trip test coverage
- **Virtual nodes**: abstract base nodes in YAML without concrete C++ classes

Adding a new backend is purely a matter of adding new Jinja2 templates.

### New artifacts to generate

| Generated file              | Purpose                                      |
|-----------------------------|----------------------------------------------|
| `slang_importer.hpp/cpp`    | Slang AST ‚Üí veriparse AST conversion visitor |
| `slang_type_mapper.hpp`     | SV type system ‚Üí veriparse node mapping      |
| `slang_node_map.hpp`        | Slang kind enum ‚Üí veriparse NodeType enum    |

### New configuration file: `slang_node_map.yaml`

A mapping table that tells ASTGen how each veriparse node corresponds
to a Slang AST node:

```yaml
AlwaysFf:
  slang_kind: ProceduralBlockKind::AlwaysFF
  slang_class: ProceduralBlockSymbol
  children_map:
    sens: senslist
    stmt: statement

Wire:
  slang_kind: SymbolKind::Net
  slang_class: NetSymbol
  properties_map:
    name: name

Reg:
  slang_kind: SymbolKind::Variable
  slang_class: VariableSymbol
  properties_map:
    name: name
```

### New `astgen.py` entry point

```python
def process_slang_importer(description_filename, mapping_filename,
                            jinja_env, header_dir, impl_dir):
    description = load_description(description_filename)
    mapping = load_description(mapping_filename)
    nodes_dict = get_nodes_dict_from_description(description)

    # Merge mapping info into nodes_dict
    for node_name, node_info in nodes_dict.items():
        node_info['slang'] = mapping.get(node_name, {})

    header_template = jinja_env.get_template("slang_importer.hpp")
    impl_template   = jinja_env.get_template("slang_importer.cpp")
    ...
```

### Implementation plan

| Phase | Task                                                        | Effort      |
|-------|-------------------------------------------------------------|-------------|
| 1     | Add slang as conda build dependency                         | 1 day       |
| 2     | Write `slang_node_map.yaml` for Verilog-2005 subset         | 1 week      |
| 3     | Write `slang_importer.hpp/cpp` Jinja2 templates             | 1-2 weeks   |
| 4     | Add `process_slang_importer()` to `astgen.py`               | 2-3 days    |
| 5     | Validate: run existing tests via Slang importer             | 1 week      |
| 6     | Add SV nodes to `verilog_ast.yaml` + `slang_node_map.yaml`  | 2-4 weeks   |
| 7     | Update Verilog generator for SV‚ÜíVerilog-2005 downgrade      | 2-3 weeks   |
| **Total** |                                                         | **~6-10 weeks** |

### Key advantages over full rewrite

- **Transformation passes are untouched** ‚ÄĒ they work on veriparse AST
  regardless of whether it came from the Bison parser or Slang
- **ASTGen remains the single source of truth** ‚ÄĒ Slang importer is just
  another generated backend
- **Incremental** ‚ÄĒ start by validating Slang importer against existing
  Verilog-2005 test suite, then add SV nodes progressively
- **Test leverage** ‚ÄĒ existing YAML round-trip tests immediately validate
  the Slang importer for all currently supported nodes
- **No risk to existing functionality** ‚ÄĒ Bison parser remains the default,
  Slang is an opt-in frontend

## Conclusion

The most pragmatic path to SystemVerilog support is:

1. **Short term**: Use Slang as a preprocessing frontend to lower SV‚ÜíVerilog-2005,
   feeding the result into the existing veriparse pipeline (weeks of effort)
2. **Medium term**: Generate a proper `SlangImporter` via the ASTGen system,
   making Slang a first-class frontend alongside the Bison parser (6-10 weeks)
3. **Long term**: Add SV-specific AST nodes incrementally, extending the
   transformation passes and generator as needed

This strategy preserves all of veriparse's unique strengths ‚ÄĒ RTL-faithful
transformations, seeded obfuscation, selective parameter preservation ‚ÄĒ while
gaining full SV elaboration from a best-in-class parser.
